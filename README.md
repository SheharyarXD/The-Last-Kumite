# THE LAST KUMITE — NES 1-Level Demo

**Michael Rivers vs. Lightning.** One round. NROM-256 (mapper 0). Verified
in FCEUX via automated, scripted play-through with live memory inspection.

## Status

Builds cleanly with zero assembler/linker errors and has been verified
end-to-end in an emulator: Title → Intro (scrolling story text) → VS
screen → Fight (working movement, punch, kick, jump, crouch, block,
special move, AI opponent, HUD, 60-second match timer) → both Win and
Lose paths → Game Over (Ron Hall thumbs-down sequence with randomized
death text) → back to Title.

Both fighters now render as real, color-distinguished 16×16 sprites
(Michael in red, Lightning in blue) built from the client's reference art
via a real, repeatable conversion pipeline (`tools/`), and the fight stage
uses a converted version of the client's castle-ruins background
(`assets/32732.png`) instead of a flat placeholder.

See `docs/testing_report.md` for the full bug list and verification
methodology, and `docs/asset_pipeline.md` for how the art conversion
pipeline works and how to re-run or extend it.

## Building

Requires the [cc65](https://cc65.github.io/) toolchain (`ca65` + `ld65`).

```bash
sudo apt-get install cc65      # Debian/Ubuntu
brew install cc65              # macOS
```

```bash
make            # builds build/TheLastKumite.nes
make run        # builds and launches in FCEUX (EMU= in the Makefile)
make clean      # removes build/
```

The CHR art (`chr/tiles.chr`) and stage background data
(`src/stage_bg.inc`) are committed/shipped already-built, so a plain `make`
does not require Pillow or re-running the art pipeline. If you change the
source art, rebuild the affected half:

```bash
make chr        # re-author + re-convert character sprites
make bg         # re-convert the fight-stage background
make assets     # both
```

`make chr`/`make bg` require Python 3 with Pillow (`pip install pillow`).

`build_rom.py` (project root) only assembles the final iNES header onto the
linked PRG + CHR data — it is not a 6502 assembler. Actual assembly is
ca65/ld65, driven by the Makefile.

## Controls

| Input | Action |
|---|---|
| D-Pad Left/Right | Move |
| D-Pad Down | Crouch |
| D-Pad Up | Jump |
| A | Punch (5 damage) |
| B | Kick (10 damage) |
| Hold Back (away from opponent) | Block |
| Down + B + A | Special move (Perfect Guard counter) |
| Start | Begin / pause / advance text |

## Match rules

- One round, 100 HP (Michael) vs. 80 HP (Lightning), 60-second timer
- Win: reduce Lightning's HP to zero
- Lose: Michael's HP reaches zero
- Draw/timeout: higher HP wins

## Known limitations

See `docs/testing_report.md` for the complete list. Headline items:

- VS-screen character portraits are solid colored blocks, not detailed
  art (this was already the case in the original source, which describes
  them as placeholders in its own comments).
- Sound effects are wired into every gameplay event and verified for
  correct assembly/linkage, but not audibly verified (the automated test
  environment runs FCEUX with audio disabled).
- The fight-stage background is a luminance-based conversion of the
  client's castle artwork, not a hand-painted NES tileset — it reads
  clearly as a castle silhouette with clouds and foliage, but is
  necessarily lower-fidelity than the source photo/illustration, which is
  an inherent limit of fitting a detailed image into 4 colors and ~96
  unique 8×8 tiles.
- Character sprites are simplified NES-proportioned silhouettes that
  follow the client reference sheets' poses and color scheme (red gi /
  blue gi), not pixel-traced copies of the photographic reference art —
  the reference sheets are higher resolution and more detailed than a
  16×16 NES sprite can represent.
